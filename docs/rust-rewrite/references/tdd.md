# 严格 TDD 纪律（移植场景）— 必读

> 本项目**绝对遵循** `/tdd` SKILL。任何写 `.rs` 的人/agent 开工前必须读完本文件。
> 违反 red→green 顺序（先写实现再补测试）的产出，按返工处理。

## 0. 为什么强调它（本项目踩过的坑）

这是 1:1 **移植**任务，Go 的实现和测试都已存在。最容易犯的错是：
"把 `foo.go` 整文件翻成 `foo.rs`，再把 `foo_test.go` 整文件翻成 `foo_test.rs`"。
这就是下面 §2 的 **horizontal slicing（横切）反模式**，产出的是 *test-after*，不是 *test-first*。
**移植不豁免 TDD**：即使 Go 测试现成，也必须**一次一个行为**地 红→绿，详见 §3「移植场景的 TDD」。

## 1. 核心哲学（来自 /tdd SKILL）

- 测试验证**行为**（通过公开接口），不验证实现细节。实现可以全改，测试不该改。
- **好测试**是集成风格：走真实代码路径、读起来像规格说明、能在重构后存活。
- **坏测试**耦合实现：mock 内部协作者、测私有方法、绕过接口从外部校验。
  警示信号：重构（行为没变）时测试却挂了 —— 那它测的是实现，不是行为。

## 2. 反模式：横切（Horizontal Slices）—— 严禁

**不要先写完所有测试再写所有实现**，也不要先写完实现再补测试。两者都是横切。

横切产出"垃圾测试"：
- 批量写的测试测的是**想象**的行为，不是**真实**行为；
- 容易退化成测"形状"（数据结构 / 函数签名）而非用户可见行为；
- 对真实变化不敏感：行为坏了它还绿，行为没坏它却红。

```
错误（横切）：
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5
（"先翻所有 impl 再翻所有 test" 也是这种）

正确（垂直切片 / tracer bullet）：
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
  ...
```

## 3. 移植场景下的 TDD（本项目专属规则）

Go 侧已有 impl + test，但仍**严格 test-first，逐行为推进**：

对目标包里的**每一个行为**（通常对应一个 Go 子用例 / 一个函数的一类输入）：

1. **RED**：从 Go `_test.go` 移植**这一个**行为的测试到 `<file>_test.rs`（`use super::*;`）。
   此刻该函数在 `.rs` 里还是 `todo!()`（带 `// DEFER`/`// blocked-by:` 或直接未实现）。
   跑 `cargo test -p tsgo_<pkg> <test_name>` → **看到它红**（编译过但断言失败，或 `todo!()` panic）。
2. **GREEN**：从 Go 移植**这一个**函数的实现，让该测试变绿。只写够让当前测试过的代码。
3. 重复下一个行为。**不要**提前把后面的函数都实现了。
4. 一个文件的所有行为绿了之后，再做 §4 重构。

> 关键纪律：**先有红的测试，才允许写让它变绿的实现**。如果你发现自己"先把实现整文件翻完了"，停下，删回到上一个绿点，按行为重来。

### 留下 red→green 证据（收口要查）

- 每个包在 `impl.md` / `tests.md` 勾选时，应能说明推进顺序是逐行为的。
- 推荐：每个行为一个小提交（`test(core): RED <behavior>` 紧接 `feat(core): GREEN <behavior>`），或在 worklog 里记录每步 red→green。本项目当前**不强制 commit**，但收口审查会问"红在哪一步看到的"。

## 4. 重构（仅在 GREEN 时）

所有相关测试绿之后，再找重构点：抽重复、深化模块（小接口/深实现）、按需 SOLID。
每步重构后立刻重跑测试。**红着的时候绝不重构** —— 先回到绿。

## 5. 每个循环自检清单

```
[ ] 测试描述行为，不是实现
[ ] 测试只用公开接口
[ ] 测试能在内部重构后存活
[ ] 实现是"刚好让当前测试通过"的最小量
[ ] 没有提前实现后面测试才需要的功能
[ ] （本项目）测试在独立 <file>_test.rs，注释英文，带 // Go: 锚，expected 取 Go 字面量
```
